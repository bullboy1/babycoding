## 課題

AIコーディングエージェントには、どれも同じ2つのコストが最初から組み込まれている。

1. **セッションごとにコードベースを学び直す** — grepし、ファイルを読み、ゼロからメンタルモデルを再構築する。このコストはリポジトリのサイズに比例して増えていく。
2. **再利用するより発明する方を選んでしまう** — 言語モデルにとっては、既存の関数が同じ仕事をすることを証明するより、新しい関数を書く方がその場では「安く」つくからだ。

この2つのコストを何百セッションも積み重ねると、同じ機能が5つの異なる実装として存在し、どれが正なのか誰も分からず、どれも削除する勇気を持てない状態になる。これは仮説ではない。ガードレールなしにAI支援で開発を続けたコードベースが必ず行き着く先だ。

**FRPは「コードベースを探索する」を「調べる」に変え、「再利用してほしい」を強制力のあるプロトコルに変える。**

## 仕組み

`CLAUDE.md`、`AGENTS.md`、あるいは使っているツールが読み込むファイルを1つ、プロジェクトルートに置くだけでいい。そのファイルはエージェントに次のように指示する。**関数を書く前に、必ずレジストリ（`FUNCTIONS.md`）を確認すること。見つかった → 再利用する。ほぼ見つかった → 拡張する。本当に存在しない → 作成し、その上でレジストリに登録する。** すべてのタスクは、そのレジストリをコードと同期させて終える。

このプロトコルは3つの層で動いており、各層は最も誠実に扱える者が担当する。

| 層 | 内容 | 担当 | 理由 |
|---|---|---|---|
| **事実（Fact）** | 関数名 / シグネチャ / パス / 呼び出し箇所数 | スキャンスクリプト（エージェント自身が書く） | 機械的な事実はスクリプトに任せるべき — 記憶のズレも、コピーミスも起きない |
| **意味（Semantic）** | L1〜L4の階層 / 一行の説明文 | エージェントが段階的に | 判断が絡む部分はスクリプト化できない。触れた行だけを注釈するので、タスクごとのコストは小さく抑えられる |
| **強制（Enforcement）** | レジストリとコードの整合性 | pre-commitフック | 「プロンプトがダメだと言っている」より「環境がダメだと言っている」方がはるかに信頼できる |

そしてこのプロトコルは、最初から重いツールを要求するのではなく、プロジェクトの成長に合わせて育っていく。

| ステージ | 関数数 | 仕組み |
|---|---|---|
| 1 | 30未満 | 手動管理のファイル1つ、ツール一切なし |
| 2 | 30〜200 | エージェントが自分でスキャンスクリプトを書く。事実列は自動化され、意味列は遅延的に注釈されたままになる |
| 3 | 200超 | レジストリをモジュール単位でシャーディングする。ルートファイルはインデックスになる。pre-commitフックがドリフトを不可能にする |

関数の階層は**重要度ではなく依存方向**によって決まる — importは下方向（L4→L3→L2→L1）にしか流れることが許されないので、逆方向のimportは重複関数と同じくプロトコル違反になる。

- **L1 — 純粋なユーティリティ。** ビジネス概念を含まず、内部importもない。
- **L2 — 共有コンポーネント。** 2つ以上の機能で使われている、あるいは明らかに使い得る。
- **L3 — ビジネスロジック。** ドメイン概念（user、order、invoiceなど）を持ち、単一責任である。
- **L4 — エントリーポイント。** ルーティング、CLIコマンド、オーケストレーション。

## 実際に何か変わるのか？測ってみた結果

15日間、実際の作業でFRPを運用した — 長時間の無人稼働による`/loop`形式のタスクキューで、Claude Opus 4.8をMax 5xシートでExtended Thinkingとサブエージェントを有効にして動かし、別トラックとしてCodex CLI + GPT-5.5-high（Plusシート）でも並行して検証した。以下の「タスク」は、開発フレームワークのタスクグラフ上の1ノードを指す。毎回、比較可能な作業単位だ。

  <img src="assets/perf-comparison.svg" alt="Before/after comparison: Codex completes 10 tasks per quota cycle instead of 6; Claude sustains a 15-day continuous run instead of collapsing after about 1 hour" width="820">

| 構成 | FRP導入前 | FRP導入後 |
|---|---|---|
| Codex CLI（GPT-5.5-high、Plusシート） — クォータが尽きるまでに完了したタスク数 | 6タスクで上限 | **10タスク** |
| Claude Opus 4.8（Max 5x、Extended Thinking、サブエージェント） — 5時間枠内での最長無人稼働 | **約1時間**、時にはそれより短い時間でノイズ化して崩壊 | **15日間**連続の`/loop`タスクを完走 |

*これらは著者本人が個人利用の中で記録した数値であり、独立した第三者によるベンチマークではない。それでも共有するのは、「トークン消費が減る」という主張を、鵜呑みにしてもらう必要はないと考えているからだ。*

**なぜこの結果が成り立つのか**: 長時間タスクにおけるエージェントのトークン予算のほとんどは、コードを書くことに使われているわけではない。すでに存在するコードを*再発見する*ことに使われている。FRPはその発見の作業を、すでにコンテキストに乗っているファイルへの参照に置き換える。だから、どのタスクもより安く始まり、劣化がより遅く、コンテキストやクォータの壁に当たるまでより長く生き延びる。

## 具体例

`src/lib`全体で約85個のエクスポート関数を持つ、中規模のバックエンドを例にとる。FRPがない場合、3週間の間に発生した3つの異なるタスクが、それぞれ「注文の割引後配送料を計算する」という同じ要件に行き着く — そしてそれぞれが別のセッションで処理され、互いの記憶を持たない。結果として`calcShipping`、`getShippingCost`、`computeShippingTotal`がすべて存在し、それぞれ丸め処理の扱いが微妙に食い違い、次のバグ報告は「チェックアウトが実際に呼んでいるのはどれだ？」になる。

FRPがある場合:

- **タスク1**が`computeShippingCost`（L3）を実装し、`FUNCTIONS.md`に登録して完了する。
- **タスク2**は同じ機能に加えてプロモコードによる割引が必要になる。エージェントはまずレジストリを検索し、**HIT**を得て、それが80%一致していると判断し、新たな兄弟関数を書く代わりに`promoCode?`パラメータを追加して**EXTEND**する。
- **タスク3**は3週間後、まったく新しいセッションで、何かを書く前に「shipping」を検索し、同じ関数を見つけ、そのまま再利用する。

関数は1つ、正解は1つ、「どれが正なのか」を突き止めるデバッグセッションはゼロ — そしてレジストリのトレース（`REGISTRY: HIT computeShippingCost (L3) …`）があるおかげで、差分を読み直さなくても、エージェントがタスクごとにどんな判断を下したかを正確に検証できる。

## クイックスタート

1. [`CLAUDE.md`](CLAUDE.md)（あるいは上表で自分のツールに対応するファイル）をプロジェクトルートにコピーする。
2. あとは普通にタスクを始めればいい。プロトコルは自動的に働く。`FUNCTIONS.md`がまだなければ最初にブートストラップし、すでにあれば新しい関数を書く前に必ず確認される。
3. 既存の古いコードベースにある関数は、事前に一括で注釈が付くわけでは**ない** — そんなことをすれば一度きりの巨大なトークン消費になってしまう。意味列は空のままで、そのタスクがその関数に触れた人によって1行ずつ埋められていく。レジストリはコールドスタートのコストをほぼゼロに抑えながら、徐々に価値を発揮していく。

## この数値である理由

- **関数は50行以内 / ファイルは300行以内 / ネストは3段まで**: モデルは「きれいに保って」のような形容詞的な指示より、はっきりした数値の方がずっと確実に守る。
- **フォルダ階層は4段まで**: 階層が深いほどファイルを見つけるための往復が増える — 分離は命名と責務によって実現するべきで、ネストによって実現するものではない。
- **Rule of Three**: 誤った抽象化は、重複よりもコストが高い。2回目の重複は許容し、3回目でようやく抽出を強制する — これによって時期尚早な抽象化が始まる前に止められる。
- **説明文は12語以内、テーブルの余白調整なし、ファイル全体の書き直しなし**: レジストリ自体が毎セッション必ず支払う固定コストだ。そこに含まれる1バイトたりとも、その存在価値を証明しなければならない。
- **必須の`REGISTRY:`トレース行**: 数十トークンのコストで2つの効果を買っている。声に出して言わなければならない手順は実際に実行される可能性が高くなること、そしてモデルがそれを省略していないかを一目で監査できることだ。

## トークンのざっくりした計算

タスクあたりの固定コストは、おおよそプロトコル本体（~1,000トークン）＋関連するレジストリのシャード（数百トークン）＋インクリメンタルな注釈（1〜200トークン）で、合計すると~1,500トークン程度になる。これを、中規模コードベースの闇雲な探索と比較してみてほしい。それだけで数万トークンに達することが常態であり、その上さらに、すでに存在していた関数を生成するためのトークンまで積み重なる。プロジェクトが大きくなり、セッション数が増えるほど、この差はさらに広がっていく。

## FAQ

**既存の、整理されていないコードベースにこれを後付けできるか？** できる。ブートストラップは事実列だけを埋める。意味列はすべて空の状態から始まり、遅延的に埋められていく。初期テーブルの構築コストは低い。

**より弱い/安価なモデルでも実際にこれを守れるか？** このプロトコルは判断が必要な場面を決定木とIF/THENルールに変換し（階層の判定は4つのyes/noチェックにすぎない）、監査可能性のために`REGISTRY:`トレース行を強制し、すべての土台をpre-commitフックで固める。これによって、知性の問題をプロセスの問題に変換している — そして安価なモデルは、判断力を発揮することより、プロセスに従うことの方がはるかに得意だ。

**安価なモデルが賢いコードを書くようになるのか？** 正直に言えば、ならない — このプロトコルが改善するのはアーキテクチャと再利用率であり、個々の関数本体の巧妙さではない。しかしこれは積み重なっていく。再利用が1回増えるごとに、弱いモデルがゼロから書かなければならない関数が1つ減る。つまり品質は、すでに検証済みのコンポーネントが増えていくライブラリの中に蓄積されていく。モデルの役割は静かに「作者」から「組み立て役」へと移っていく — これはまさに、安価なモデルが確実に得意な仕事だ。

**複数人、あるいは複数セッションが同時に作業する場合はどうなるか？** レジストリはgit管理下にあるので、コンフリクトは通常のマージで解決する。ステージ2以降では、スキャンスクリプトを再実行すると事実列はゼロから再構築されるが、契約上、手動で書いた意味列は*保持*することが求められる。

**レジストリ自体が死んだエントリの山になってしまわないか？** ならない — `refs`列によって孤立した関数が可視化される。参照数がゼロになると`[DEPRECATED]`が付き、次に誰かがそのモジュールに触れたときに（関数と行の両方が）削除される。エントリは、入ってきたときと同じ経路で去っていく。

## 調整可能なパラメータ

ステージの閾値（30 / 200）、行数制限（50 / 300）、フォルダ階層（4）、説明文の長さ（12語）はすべて単なる数値であり、チームの好みに合わせて変更してかまわない。変更する場合は、プロトコルファイルの§10にあるセルフチェックリストも合わせて更新すること。ルールとチェックが食い違わないようにするためだ。
